# Навык «Базы данных» (Техническая спецификация)

## 1. Схема базы данных «View Ridge Gallery»

Агент использует следующие определения таблиц для ЛР №3–8:
- **ARTIST:** `ArtistID` (PK, Serial), `Name`, `Nationality`, `BirthDate`, `DeceasedDate`.
- **WORK:** `WorkID` (PK, Serial), `Title`, `Description`, `Copy`, `ArtistID` (FK).
- **CUSTOMER:** `CustomerID` (PK, Serial), `Name`, `AreaCode`, `PhoneNumber`, `SecondName` (доп. поле).
- **TRANSACTION:** `TransactionID` (PK, Serial), `DateAcquired`, `AcquisitionPrice`, `AskingPrice`, `SalesPrice`, `PurchaseDate`, `CustomerID` (FK), `WorkID` (FK).
- **CUSTOMER_ARTIST_INT:** Таблица пересечения. `ArtistID` (FK), `CustomerID` (FK).

## 2. Детальный реестр лабораторных работ

### ЛР №1: Развертывание среды
- СУБД: PostgreSQL 15+.
- Задачи: Создание БД `test`, роли `test_user`, настройка прав (`GRANT ALL PRIVILEGES`).
- PgAdmin: Использование ERD Tool для визуального проектирования.

### ЛР №2: Основы SQL и Транзакции
- Таблицы: `courses` (c_no, title), `students` (s_id, name), `exams` (score).
- Запросы: `INNER JOIN`, `LEFT JOIN`, `GROUP BY` с `HAVING`.
- Транзакции: Демонстрация изоляции через два окна терминала.

### ЛР №3: Проектирование схемы (DDL)
- **Задачи:** Создание таблиц галереи.
- **PK:** Демонстрация создания первичного ключа через `CREATE TABLE` и через `ALTER TABLE ADD PRIMARY KEY`.
- **Последовательности:** `CREATE SEQUENCE CustID START 1000`.

### ЛР №4: Целостность и оптимизация
- **Связи:** Внедрение `ON DELETE CASCADE` для внешних ключей.
- **Оптимизация:** Создание индексов `CREATE INDEX idx_artist_name ON ARTIST(Name)`.
- **CHECK:** Ограничение `CHECK (SalesPrice > AcquisitionPrice)`.

### ЛР №5: Процедурное программирование (Курсоры)
- **Объект:** Процедура `CustomerInsert`.
- **Логика:** Принимает `artistNationality`. Открывает `CURSOR` для выборки всех художников этой нации. В цикле `FOR ... LOOP` связывает их с новым клиентом.

### ЛР №6: Управление транзакциями (PL/pgSQL)
- **Объект:** Процедура `NewCustomerWithTransaction`.
- **Логика:** Проверяет наличие свободной картины (`CustomerID IS NULL`). Если найдена ровно ОДНА запись — подтверждает покупку. Если 0 или >1 — делает `ROLLBACK` с уведомлением `RAISE NOTICE`.

### ЛР №7: Триггеры автоматизации
- **BEFORE Trigger:** Расчет `AskingPrice = AcquisitionPrice * 2`.
- **AFTER Trigger:** При вставке в `WORK` автоматически создавать "пустую" запись в `TRANSACTION` с `current_date`.

### ЛР №8: Безопасность и представления
- **VIEW:** Создание `CustomerPurchases` с алиасами для совпадающих имен.
- **INSTEAD OF Trigger:** Триггер `Title_Update`, позволяющий изменять название картины прямо через View (перенаправляет `UPDATE` на таблицу `WORK`).
- **EXCEPTION:** Блок `EXCEPTION WHEN others THEN` с использованием `GET STACKED DIAGNOSTICS`.

## 4. Протокол экспертной валидации и контроля качества

После генерации каждой работы агент обязан проверить результат по следующим критериям:

1. **Глубина контента (Критично):**
   - Объем текстового отчета должен быть увеличен в 2 раза относительно базовых шаблонов (минимум 5000-6000 знаков).
   - Обязательно подробное теоретическое описание каждой используемой команды (не просто "создал таблицу", а объяснение зачем нужен каждый тип данных, как работает индекс или почему выбран конкретный триггер).
   - Развернутые выводы по каждому этапу работы.

2. **Объем и детализация промптов:**
   - Промпт для изображения не может быть короче 1000 символов.
   - Описание должно включать: точное положение курсора, состояние дерева объектов в pgAdmin, цветовые схемы синтаксиса в Query Tool, ASCII-границы в терминале psql.

3. **Синхронизация данных (Zero Contradiction):**
   - Если в SQL-коде отчета создается клиент "Иванов" с ID 1001, то в промпте для скриншота этого результата ДОЛЖЕН быть указан именно "Иванов" и ID 1001.
   - Любые вычисляемые данные (например, `AskingPrice = AcquisitionPrice * 2`) в промпте должны соответствовать математике из отчета. 
   - Названия баз данных, схем и таблиц в промптах должны строго совпадать с кодом.

4. **Структурная проверка:**
   - Наличие всех файлов в `source/` (скрипты, промпты).
   - Соответствие оформления ГОСТ 7.32-2017 (шрифт, интервалы, подписи).
