# Лабораторная работа №2. Отчет
**Дисциплина:** Технологии баз данных
**Тема:** Проектирование схемы реляционной базы данных, манипулирование данными и транзакционная целостность.

## 1. Введение
Целью выполнения второй лабораторной работы является закрепление теоретических знаний о реляционной алгебре посредством написания SQL-запросов в диалекте PostgreSQL. В фокусе внимания находились задачи создания связанных таблиц (DDL), наполнения их данными (DML), агрегации информации, объединения нескольких отношений через JOIN, а также управление транзакциями для обеспечения логической целостности данных.

## 2. Ход выполнения работы

### 2.1. Описание DDL и проектирование структуры
Была спроектирована микро-система учета успеваемости. Отношения нормализованы до 3НФ. 
- Таблица `subject` (Учебные дисциплины) со строковым первичным ключом.
- Таблица `scholar` (Учащиеся) с автоинкрементным суррогатным ключом (`SERIAL`).
- Таблица `testing` (Результаты тестирования), выполняющая роль связующей таблицы (М:М) с композитным первичным ключом и двумя внешними ключами.

```sql
CREATE TABLE subject (
    sub_id VARCHAR(15) PRIMARY KEY,
    name VARCHAR(150),
    lec_hours INTEGER
);

CREATE TABLE scholar (
    stud_id SERIAL PRIMARY KEY,
    fullname VARCHAR(100),
    enrollment_year INTEGER
);

CREATE TABLE testing (
    stud_id INTEGER REFERENCES scholar(stud_id),
    sub_id VARCHAR(15) REFERENCES subject(sub_id),
    grade INTEGER,
    PRIMARY KEY(stud_id, sub_id)
);
```

### 2.2. Заполнение таблиц данными
Для инициализации БД выполнены команды `INSERT INTO`. В таблицу `scholar` имена и года зачисления добавлялись с указанием столбцов, так как `stud_id` генерировался автоматически благодаря типу `SERIAL`.

### 2.3. Формирование запросов (DQL)
Для извлечения аналитической информации применялись конструкции `SELECT`:
1. Получение списка всех уникальных годов поступления: 
   `SELECT DISTINCT enrollment_year FROM scholar;`
2. **Использование INNER JOIN:** Извлечение списка студентов и их оценок по конкретному предмету (код `SUB1`). Связь установлена по полю `stud_id`.
   ```sql
   SELECT sc.fullname, t.grade 
   FROM scholar sc 
   INNER JOIN testing t ON sc.stud_id = t.stud_id 
   WHERE t.sub_id = 'SUB1';
   ```
3. **Использование LEFT OUTER JOIN и агрегации:** Подсчет количества сданных экзаменов по каждому предмету. Использована группировка `GROUP BY` и фильтр по результату функции агрегации `HAVING COUNT(...) > 0`.
   ```sql
   SELECT s.name, COUNT(t.grade) AS exam_count 
   FROM subject s 
   LEFT OUTER JOIN testing t ON s.sub_id = t.sub_id 
   GROUP BY s.name 
   HAVING COUNT(t.grade) > 0;
   ```

### 2.4. Обеспечение целостности через транзакции
Был смоделирован процесс реорганизации: создание справочника учебных групп и назначение группы существующим студентам. Поскольку эти действия не имеют смысла друг без друга, они были обернуты в блок транзакции. 
```sql
CREATE TABLE stud_groups (
    group_code VARCHAR(15) PRIMARY KEY,
    monitor_id INTEGER NOT NULL
);
ALTER TABLE scholar ADD COLUMN group_code VARCHAR(15);

BEGIN;
INSERT INTO stud_groups VALUES ('B-202', 1);
UPDATE scholar SET group_code = 'B-202';
COMMIT;
```
В случае возникновения сбоя на этапе `UPDATE`, можно было бы использовать оператор `ROLLBACK` для отката вставки в `stud_groups`, тем самым предотвратив появление "повисших" групп без студентов.

### 2.5. Виртуальные таблицы (Представления)
Для удобного формирования отчета об отличниках было создано постоянное представление `excellent_grades`, которое инкапсулирует в себе трехтабличное объединение.
