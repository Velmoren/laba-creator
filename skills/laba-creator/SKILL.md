# ГЕНЕРАЛЬНЫЙ СТАНДАРТ «LABA-CREATOR»

Этот навык является головным. Он определяет общие правила генерации всех типов работ в проекте.

## 1. Диспетчеризация тем
При поступлении запроса на генерацию, использовать следующие специализированные навыки:
- **Операционные системы реального времени (FreeBSD):** `osrv-labs`
- **Технологии баз данных (SQL/NoSQL):** `db-tech-labs`
- **Схемотехника и СТУ (Micro-Cap):** `schematic-labs`

## 2. Стандарты работы с изображениями (Скриншотами)

### 2.1. Правило «Листинг = Скриншот»
**Критическое требование:** Любой значимый блок кода, листинг команд терминала или вывод программы в отчете должен быть заменен или продублирован изображением. Текстовые блоки ` ```bash ` допустимы только для мелких связующих команд (например, `cd`).

### 2.2. Оформление в документе
Генератор (`generate_labs.js`) автоматически обрабатывает теги `![Описание](файл.png)` по следующим правилам:
1.  **Первое вхождение:** Вставляется изображение (центровка, ширина ~13см) + подпись под ним: *«Скриншот N – Описание»*.
2.  **Повторное вхождение:** Если тот же файл изображения встречается в отчете еще раз, он заменяется текстовой ссылкой: *(см. Скриншот N)*.
3.  **Нумерация:** Скриншоты нумеруются автоматически в рамках одного документа.

### 3.2. Специфика для Micro-Cap 9 (Схемотехника)
- **Инструмент:** В промпте ОБЯЗАТЕЛЬНО указывать программу "Micro-Cap 9".
- **Масштаб:** Окно программы должно занимать 100% изображения (без фона рабочего стола, иконок и панелей задач).
- **Визуальная целостность:** Интерфейс программы должен быть ИДЕНТИЧЕН на всех скриншотах одного студента. Недопустимо смешивать темную и светлую темы в одной работе. 
- **Детализация темы:** Четко прописывать цвета: фон рабочего поля (Workspace), цвет сетки (Grid), цвета линий графиков и шрифтов. Указывать стиль окна (классический Windows 98/XP для Micro-Cap 9).
- **Интерфейс:** Упоминать характерные элементы: меню "Analysis", панель инструментов со значками [LIN], [N], курсоры.

## 4. Контрольные вопросы
- **Формат:** Ответы на вопросы НЕ включаются в основной файл отчета.
- **Файл:** Рядом с отчетом создается отдельный файл **`Ответы на контрольные вопросы.md`**.
- **Структура:** Весь список вопросов из методички приводится полностью в формате:
  1. Вопрос
  Ответ
  2. Вопрос
  Ответ
- **Полнота:** Должны быть даны ответы на ВСЕ вопросы из списка методички (правило "одного номера" отменяется).

## 4. Технические стандарты (Файловая система)

### 4.1. Правило папки `source`
Все технические результаты работы (скрипты `.sh`, исходники `.c`, файлы баз данных, схемы `.cir`, архивы) ДОЛЖНЫ располагаться в подкаталоге **`source/`** внутри папки лабораторной работы. Корень папки ЛР зарезервирован только для документации (`.md`, `.txt`, `.png`).

### 4.2. Уникальность контента
Запрещено использовать одинаковые названия папок и файлов для разных студентов. 
- Студент А: `workspace/project_01`, `run.sh`
- Студент Б: `lab_work/fs_task`, `start.sh`
Названия должны генерироваться на основе метаданных студента.

### 4.3. Материализация
Агент обязан физически создавать все структуры в `source/`, которые он описывает в отчете. Отчет должен быть 100% правдив относительно состояния файлов.
