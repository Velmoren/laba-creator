# ГЕНЕРАЛЬНЫЙ СТАНДАРТ «LABA-CREATOR»

Этот навык является головным. Он определяет общие правила генерации всех типов работ в проекте.

## 1. Конфигурация метаданных студента
Для каждого студента создается файл **`student_settings.json`**. Данные из него являются приоритетными.

## 2. Диспетчеризация тем (Делегирование задач)
При поступлении запроса на генерацию лабораторных работ, агент должен активировать специализированный навык в зависимости от дисциплины:
- **Операционные системы реального времени (FreeBSD):** Активировать скилл `osrv-labs`
- **Технологии баз данных (SQL/NoSQL):** Активировать скилл `db-tech-labs`
- **Схемотехника и СТУ (Micro-Cap):** Активировать скилл `schematic-labs`
- **Курсовые работы (Проектирование БД):** Активировать скилл `course-work`

## 3. Стандарты содержания (Отчеты)
- **Объем:** Минимальный объем текстового контента — **2 страницы формата А4** (после генерации в DOCX). Это обязательное требование.
- **Структура:** 
    1. Расширенное введение (архитектура ОС, принципы работы изучаемого узла).
    2. Пошаговое выполнение с подробным описанием каждой команды и её флагов.
    3. Глубокий технический анализ результатов (сравнение характеристик, объяснение причин именно такого поведения системы).
    4. Объемное заключение.
- **Визуализация:** Все текстовые листинги должны быть заменены скриншотами, ПРИ УСЛОВИИ наличия соответствующего промпта. Если изображение повторяется, используется перекрестная ссылка `(см. Скриншот N)`.

## 3. Стандарты визуализации (Промпты)
- **Детализация:** Промпт должен исключать любую двусмысленность.
- **Интерфейс:** Описывать цвета, шрифты, размер окна, положение курсора, цветовую схему синтаксиса.
- **Вывод (Output):** Описывать не только вводимую команду, но и **полный ответ системы**.
- **ОС:** Строгое соответствие `os` и `ui_theme` из настроек студента.

## 5. Обязательная валидация (Чек-лист перед сдачей)
После завершения генерации агент обязан проверить:
1.  **Наполненность отчетов (Критично):** Суммарный объем текста в отчете должен гарантировать минимум 2 страницы в Word. При нехватке — расширить теоретические блоки и анализ.
2.  **Наполненность промптов (Анти-халтура):** Промпты для скриншотов не должны быть пустыми заглушками или содержать один и тот же скопированный текст. Агент обязан проверить содержимое файлов `.txt`, чтобы убедиться в наличии детального описания интерфейса и точного вывода терминала/программы.
3.  **Соответствие промптов:** Соответствуют ли промпты тексту отчета и названиям ожидаемых картинок `![...]`?
4.  **Source:** Все ли описанные файлы физически созданы в папке `source/` и содержат ли они реальный код/данные?
5.  **Инструментальная верификация:** Выполнить `ls -R` или `find` для подтверждения наличия всех структур в `source/`.
6.  **Логика замены кода на скрины:** 
    - Если для блока кода ЕСТЬ промпт -> заменить код ссылкой на изображение `![...] (screenshot_X.png)`.
    - Если для блока кода НЕТ промпта -> ОСТАВИТЬ блок кода текстом ` ```bash `.
    - Не допускать удаления кода без замены его визуальным подтверждением.
