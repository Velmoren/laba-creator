# СОДЕРЖАНИЕ

# ВВЕДЕНИЕ {#введение .Heading-1}

Современный мир информационных технологий невозможен без использования
баз данных. Все автоматизированные информационные системы связаны с
функциями долговременного хранения и обработки информации, которая
является ключевым фактором, определяющим эффективность любой сферы
деятельности.

В настоящее время сфера дополнительного профессионального образования
переживает период активного роста. Компьютерные курсы являются одним из
наиболее востребованных направлений обучения, позволяющим специалистам
различных возрастов и профессий осваивать новые технологии и получать
актуальные знания. Эффективное управление учебным процессом требует
автоматизации множества операций: от учёта слушателей и преподавателей
до формирования расписания и отчётности.

Актуальность темы курсовой работы обусловлена тем, что автоматизация
деятельности образовательного центра позволяет существенно повысить
качество управления учебным процессом, снизить временные затраты на
рутинные операции, минимизировать ошибки при обработке данных и
обеспечить оперативный доступ к актуальной информации как для
администрации, так и для обучающихся.

Целью курсовой работы является проектирование и реализация базы данных
для автоматизированной информационной системы управления образовательным
центром «Компьютерные курсы».

Для достижения поставленной цели необходимо решить следующие задачи:

1\. Провести системный анализ предметной области и описать объект
автоматизации

2\. Изучить существующие аналоги и обосновать выбор направления
разработки

3\. Спроектировать инфологическую и даталогическую модели базы данных

4\. Выполнить нормализацию схемы базы данных

5\. Реализовать базу данных в СУБД PostgreSQL

6\. Разработать представления, триггеры и механизмы обеспечения
безопасности

Объектом исследования является образовательный центр по проведению
компьютерных курсов.

Предметом исследования --- процессы хранения и обработки данных в
автоматизированной системе управления учебным процессом.

При выполнении работы использовались методы системного анализа,
моделирования данных, методы нормализации реляционных баз данных.

Курсовая работа состоит из введения, трёх глав, заключения, списка
использованных источников и приложений.

### 1.2.1. CASE-средства для проектирования БД {#case-средства-для-проектирования-бд .Heading-3}

Для проектирования базы данных могут использоваться специализированные
CASE-средства (Computer-Aided Software Engineering), которые позволяют
автоматизировать процесс разработки моделей данных \[2, 10\]. Рассмотрим
наиболее популярные инструменты:

**ERwin Data Modeler** --- один из наиболее популярных инструментов для
проектирования баз данных. Позволяет создавать инфологические и
даталогические модели, генерировать SQL-скрипты для различных СУБД,
проводить прямой и обратный инжиниринг.

**PowerDesigner** --- мощное средство моделирования данных от SAP,
поддерживающее различные нотации и позволяющее работать с большими
проектами.

**DBDesigner** --- бесплатное средство для визуального проектирования
баз данных.

**pgModeler** --- специализированное средство для проектирования баз
данных PostgreSQL с открытым исходным кодом.

Для данного проекта рекомендуется использовать **ERwin Data Modeler**
благодаря широким возможностям и поддержке нотации Crow\'s Foot, которая
наиболее наглядно отражает связи между сущностями.

### 1.2.2. Классы СУБД {#классы-субд .Heading-3}

Рассмотрим основные классы систем управления базами данных, подробно
описанные в литературе \[3, 9\]:

**Иерархические СУБД** --- организуют данные в виде дерева с отношениями
«родитель-потомок». Применяются в специализированных системах (файловые
системы, XML-базы). Примеры: IBM IMS, Windows Registry.

**Сетевые СУБД** --- развитие иерархической модели, позволяют связи
«многие ко многим» через сетевые структуры. Используются в системах с
сложными взаимосвязями. Примеры: ADABAS, Cache.

**Реляционные СУБД** --- наиболее распространённый тип, организует
данные в таблицах со связями через общие ключи. Преимущества:
теоретическая база (реляционная алгебра), гибкость запросов, стандартный
язык SQL. Примеры: PostgreSQL, MySQL, Oracle, MS SQL Server.

**Объектно-ориентированные СУБД** --- хранят данные в виде объектов, как
в ООП. Применяются для сложных структур данных. Примеры: ObjectDB, db4o.

Для проектирования базы данных образовательного центра наиболее подходит
**реляционная модель данных**, поскольку она:

• Обладает развитым математическим аппаратом

• Поддерживается всеми современными СУБД

• Позволяет эффективно обрабатывать структурированные данные

• Имеет стандартизированный язык запросов SQL

### 1.3.3. Обоснование выбора направления разработки {#обоснование-выбора-направления-разработки .Heading-3}

На основании проведённого анализа существующих систем можно сделать
вывод, что большинство решений либо слишком дороги для небольших

### 2.1.2. Связи между сущностями {#связи-между-сущностями .Heading-3}

через внешние ключи в реляционной модели. Рассмотрим основные связи
между сущностями. Типы связей определяют кардинальность взаимодействия:
\"один к одному\" (1:1), \"один ко многим\" (1:N) и \"многие ко многим\"
(M:N). Правильное определение связей обеспечивает целостность данных и
эффективность запросов. В таблице представлены основные связи между
сущностями.

          Группа                                   ведёт несколько групп

      Курс --- Группа               1:N            Один курс может иметь
                                                      несколько групп

    Группа --- Занятие              1:N              Одна группа имеет

## 2.2. Обоснование выбора модели данных {#обоснование-выбора-модели-данных .Heading-2}

Для разрабатываемой базы данных выбрана **реляционная модель данных**
\[3, 8\] по следующим причинам:

1\. **Теоретическая база.** Реляционная модель основана на
математической теории отношений, что обеспечивает строгость и
непротиворечивость данных \[17\].

2\. **Стандартизация.** Язык SQL является стандартом для работы с
реляционными базами данных, что обеспечивает переносимость и унификацию
запросов \[4, 13\].

3\. **Гибкость запросов.** Реляционная алгебра позволяет формулировать
сложные запросы к данным из множества таблиц.

4\. **Целостность данных.** Реляционная модель поддерживает механизмы
обеспечения целостности через первичные и внешние ключи \[6\].

5\. **Широкое распространение.** Большинство современных СУБД
поддерживают реляционную модель, что обеспечивает доступность
инструментов и специалистов.

6\. **Соответствие предметной области.** Данные образовательного центра
естественно представляются в виде таблиц с отношениями между ними.

Для реализации выбрана **СУБД PostgreSQL** \[7, 16\] ---
объектно-реляционная система с открытым исходным кодом, которая
соответствует реляционной модели и поддерживает все необходимые функции
для данного проекта.

## 2.4. Нормализация схемы БД {#нормализация-схемы-бд .Heading-2}

### 2.4.1. Приведение к нормальным формам {#приведение-к-нормальным-формам .Heading-3}

Нормализация схемы базы данных является важным этапом проектирования и
позволяет устранить избыточность данных \[18\]. Все таблицы базы данных
приведены к **третьей нормальной форме (3НФ)** \[3, 10\]:

**Проверка 1НФ:**

• Все атрибуты содержат атомарные значения

• Нет повторяющихся групп данных

• Каждая таблица имеет уникальный идентификатор

**Проверка 2НФ:**

• Таблицы находятся в 1НФ

• Нет частичных зависимостей неключевых атрибутов от составного ключа

• Каждая таблица имеет простой первичный ключ (одно поле SERIAL)

**Проверка 3НФ:**

• Таблицы находятся в 2НФ

• Нет транзитивных зависимостей неключевых атрибутов от первичного ключа

• Все неключевые атрибуты зависят только от первичного ключа

### 2.4.2. Обеспечение целостности {#обеспечение-целостности .Heading-3}

**Первичные ключи:** Каждая таблица имеет уникальный идентификатор
(id\_\*), генерируемый через SERIAL.

**Внешние ключи:** Все связи между таблицами реализованы через
ограничения FOREIGN KEY с каскадным удалением (ondeleteCASCADE) для
связанных записей.

**Ограничения CHECK:**

• Оценка в таблице Успеваемость ограничена значениями от 1 до 5

• Стоимость курса должна быть положительной

**Уникальные ограничения:**

• Номер договора должен быть уникальным

• Номер сертификата должен быть уникальным

**Комбинированные ограничения:**

• UNIQUE(id_клиента, id_занятия) в таблицах Посещаемость и Успеваемость
--- один клиент может иметь только одну запись посещаемости и одну
оценку на одно занятие

### 2.4.3. Устранённые аномалии {#устранённые-аномалии .Heading-3}

В результате нормализации устранены следующие аномалии:

**Аномалия вставки:** Нет необходимости дублировать информацию о курсе
или преподавателе при добавлении новой группы --- данные хранятся в
отдельных таблицах и связываются через внешние ключи.

**Аномалия удаления:** При удалении группы данные о курсе и
преподавателе сохраняются в системе.

**Аномалия обновления:** Изменение стоимости курса или данных
преподавателя автоматически отражается во всех связанных записях
благодаря нормализованной структуре.

## 3.4. Разработка форм {#разработка-форм .Heading-2}

Формы являются основным интерфейсом для взаимодействия пользователей с
базой данных \[6\]. Для работы с базой данных разработаны следующие
формы:

### 3.4.1. Форма «Клиенты» {#форма-клиенты .Heading-3}

Назначение: Добавление и редактирование информации о клиентах.

Поля формы:

• Фамилия (текстовое поле, обязательно)

• Имя (текстовое поле, обязательно)

• Отчество (текстовое поле)

• Дата рождения (поле выбора даты)

• Телефон (текстовое поле с маской)

• Email (текстовое поле с валидацией)

• Адрес (многострочное поле)

Кнопки: «Сохранить», «Отмена», «Поиск»

### 3.4.2. Форма «Запись на курс» {#форма-запись-на-курс .Heading-3}

Назначение: Регистрация нового клиента и запись на курс.

Поля формы:

• Выбор клиента (из справочника или создать нового)

• Выбор курса (из справочника)

• Выбор группы (из существующих или создать новую)

• Дата заключения договора (автозаполнение текущей датой)

• Сумма договора (автоматический расчёт)

Кнопки: «Создать договор», «Отмена», «Печать договора»

### 3.4.3. Форма «Успеваемость» {#форма-успеваемость .Heading-3}

Назначение: Ввод и проверка домашних заданий, выставление оценок.

Поля формы:

• Выбор группы

• Выбор занятия

• Список слушателей группы

• Оценка (выпадающий список 1-5)

• Домашнее задание (текстовое поле)

• Флаг «Проверено»

Кнопки: «Сохранить», «Отмена»

### 3.4.4. Форма «Посещаемость» {#форма-посещаемость .Heading-3}

Назначение: Отметка присутствия/отсутствия на занятии.

Поля формы:

• Выбор группы

• Выбор даты занятия

• Таблица слушателей с флагами присутствия

• Причина отсутствия (текстовое поле, заполняется при отсутствии)

Кнопки: «Сохранить», «Отмена»

## 3.5. Разработка отчётов {#разработка-отчётов .Heading-2}

Отчёты представляют собой документы, имитирующие внутренние документы
фирмы \[1\]. Разработаны следующие виды отчётов:

### 3.5.1. Отчёт «Список группы» {#отчёт-список-группы .Heading-3}

Назначение: Формирование списка слушателей учебной группы.

Структура отчёта:

• Заголовок: «Список учебной группы «\[название группы\]»»

• Подзаголовок: Курс: \[название курса\], Преподаватель: \[ФИО
преподавателя\]

• Таблица: № \| ФИО слушателя \| Телефон \| Email

• Итого: общее количество слушателей

Периодичность: По запросу (перед началом занятий)

### 3.5.2. Отчёт «Расписание занятий» {#отчёт-расписание-занятий .Heading-3}

Назначение: Планирование учебного процесса.

Структура отчёта:

• Заголовок: «Расписание занятий группы «\[название\]»»

• Период обучения: с \[дата\] по \[дата\]

• Таблица: № \| Дата \| Время \| Аудитория \| Тема занятия

• Подпись: Преподаватель

Периодичность: Еженедельно

### 3.5.3. Отчёт «Ведомость успеваемости» {#отчёт-ведомость-успеваемости .Heading-3}

Назначение: Документирование результатов обучения.

Структура отчёта:

• Заголовок: «Ведомость успеваемости»

• Группа: \[название\]

• Курс: \[название\]

• Период: с \[дата\] по \[дата\]

• Таблица: № \| ФИО \| Оценка за курс \| Примечание

• Подписи: Преподаватель, Руководитель

Периодичность: По окончании курса

### 3.5.4. Отчёт «Финансовый отчёт» {#отчёт-финансовый-отчёт .Heading-3}

Назначение: Учёт финансовых поступлений.

Структура отчёта:

• Заголовок: «Финансовый отчёт за \[месяц/год\]»

• Таблица: № \| № договора \| Клиент \| Сумма \| Дата оплаты \| Способ
оплаты

• Итого: Общая сумма, Наличные, Безналичные

Периодичность: Ежемесячно

### 3.5.5. Отчёт «Отчёт о задолженностях» {#отчёт-отчёт-о-задолженностях .Heading-3}

Назначение: Контроль оплат.

Структура отчёта:

• Заголовок: «Отчёт о задолженностях на \[дата\]»

• Таблица: № \| Клиент \| № договора \| Сумма задолженности

• Итого: Общая сумма задолженности

Периодичность: Еженедельно

## 3.6. Реализация ограничений и автоматизация (триггеры) {#реализация-ограничений-и-автоматизация-триггеры .Heading-2}

Триггеры представляют собой механизм автоматизации обработки данных в БД
\[5, 14\]. Реализованы следующие триггеры:

### 3.6.1. Триггер 1: Автоматический расчёт общей суммы оплат по договору {#триггер-1-автоматический-расчёт-общей-суммы-оплат-по-договору .Heading-3}

+:----------------------------------------------------------------------+
| CREATE OR REPLACE FUNCTION fn_обновить_статус_оплаты()                |
|                                                                       |
| RETURNS TRIGGER AS \$\$                                               |
|                                                                       |
| BEGIN                                                                 |
|                                                                       |
| IF NEW.сумма \> 0 THEN                                                |
|                                                                       |
| UPDATE Оплата                                                         |
|                                                                       |
| SET статус = \'Оплачен\'                                              |
|                                                                       |
| WHERE id_оплаты = NEW.id_оплаты;                                      |
|                                                                       |
| END IF;                                                               |
|                                                                       |
| RETURN NEW;                                                           |
|                                                                       |
| END;                                                                  |
|                                                                       |
| \$\$ LANGUAGE plpgsql;                                                |
|                                                                       |
| CREATE TRIGGER trg_обновить_статус_оплаты                             |
|                                                                       |
| AFTER INSERT ON Оплата                                                |
|                                                                       |
| FOR EACH ROW EXECUTE FUNCTION fn_обновить_статус_оплаты();            |
+-----------------------------------------------------------------------+

### 3.6.2. Триггер 2: Проверка доступности мест в группе {#триггер-2-проверка-доступности-мест-в-группе .Heading-3}

+:----------------------------------------------------------------------+
| CREATE OR REPLACE FUNCTION fn_проверить_место_в_группе()              |
|                                                                       |
| RETURNS TRIGGER AS \$\$                                               |
|                                                                       |
| DECLARE                                                               |
|                                                                       |
| текущее_количество INTEGER;                                           |
|                                                                       |
| максимум_группы INTEGER;                                              |
|                                                                       |
| BEGIN                                                                 |
|                                                                       |
| SELECT COUNT(\*), г.максимальный_размер                               |
|                                                                       |
| INTO текущее_количество, максимум_группы                              |
|                                                                       |
| FROM Договор д                                                        |
|                                                                       |
| JOIN Группа г ON д.id_группы = г.id_группы                            |
|                                                                       |
| WHERE д.id_группы = NEW.id_группы AND д.статус = \'Действует\'        |
|                                                                       |
| GROUP BY г.максимальный_размер;                                       |
|                                                                       |
| IF текущее_количество \>= максимум_группы THEN                        |
|                                                                       |
| RAISE EXCEPTION \'В группе нет свободных мест! Максимум: %\',         |
| максимум_группы;                                                      |
|                                                                       |
| END IF;                                                               |
|                                                                       |
| RETURN NEW;                                                           |
|                                                                       |
| END;                                                                  |
|                                                                       |
| \$\$ LANGUAGE plpgsql;                                                |
|                                                                       |
| CREATE TRIGGER trg_проверить_место_в_группе                           |
|                                                                       |
| BEFORE INSERT ON Договор                                              |
|                                                                       |
| FOR EACH ROW EXECUTE FUNCTION fn_проверить_место_в_группе();          |

# Приложение Б. Скрипт триггеров {#приложение-б.-скрипт-триггеров .Heading-1}

## Триггер 1: Автоматический расчёт общей суммы оплат по договору {#триггер-1-автоматический-расчёт-общей-суммы-оплат-по-договору-1 .Heading-2}

+:----------------------------------------------------------------------+
| \-- Функция для автоматического обновления статуса оплаты             |
|                                                                       |
| CREATE OR REPLACE FUNCTION fn_обновить_статус_оплаты()                |
|                                                                       |
| RETURNS TRIGGER AS \$\$                                               |
|                                                                       |
| BEGIN                                                                 |
|                                                                       |
| IF NEW.сумма \> 0 THEN                                                |
|                                                                       |
| UPDATE Оплата                                                         |
|                                                                       |
| SET статус = \'Оплачен\'                                              |
|                                                                       |
| WHERE id_оплаты = NEW.id_оплаты;                                      |
|                                                                       |
| END IF;                                                               |
|                                                                       |
| RETURN NEW;                                                           |
|                                                                       |
| END;                                                                  |
|                                                                       |
| \$\$ LANGUAGE plpgsql;                                                |
|                                                                       |
| \-- Создание триггера                                                 |
|                                                                       |
| CREATE TRIGGER trg_обновить_статус_оплаты                             |
|                                                                       |
| AFTER INSERT ON Оплата                                                |
|                                                                       |
| FOR EACH ROW EXECUTE FUNCTION fn_обновить_статус_оплаты();            |
+-----------------------------------------------------------------------+

**Назначение:** Автоматически устанавливает статус «Оплачен» при
добавлении новой оплаты с положительной суммой.

**Время срабатывания:** AFTER INSERT

## Триггер 2: Проверка доступности мест в группе {#триггер-2-проверка-доступности-мест-в-группе-1 .Heading-2}

+:----------------------------------------------------------------------+
| \-- Функция для проверки наличия свободных мест                       |
|                                                                       |
| CREATE OR REPLACE FUNCTION fn_проверить_место_в_группе()              |
|                                                                       |
| RETURNS TRIGGER AS \$\$                                               |
|                                                                       |
| DECLARE                                                               |
|                                                                       |
| текущее_количество INTEGER;                                           |
|                                                                       |
| максимум_группы INTEGER;                                              |
|                                                                       |
| BEGIN                                                                 |
|                                                                       |
| SELECT COUNT(\*), г.максимальный_размер                               |
|                                                                       |
| INTO текущее_количество, максимум_группы                              |
|                                                                       |
| FROM Договор д                                                        |
|                                                                       |
| JOIN Группа г ON д.id_группы = г.id_группы                            |
|                                                                       |
| WHERE д.id_группы = NEW.id_группы AND д.статус = \'Действует\'        |
|                                                                       |
| GROUP BY г.максимальный_размер;                                       |
|                                                                       |
| IF текущее_количество \>= максимум_группы THEN                        |
|                                                                       |
| RAISE EXCEPTION \'В группе нет свободных мест! Максимум: %\',         |
| максимум_группы;                                                      |
|                                                                       |
| END IF;                                                               |
|                                                                       |
| RETURN NEW;                                                           |
|                                                                       |
| END;                                                                  |
|                                                                       |
| \$\$ LANGUAGE plpgsql;                                                |
|                                                                       |
| \-- Создание триггера                                                 |
|                                                                       |
| CREATE TRIGGER trg_проверить_место_в_группе                           |
|                                                                       |
| BEFORE INSERT ON Договор                                              |
|                                                                       |
| FOR EACH ROW EXECUTE FUNCTION fn_проверить_место_в_группе();          |
+-----------------------------------------------------------------------+

**Назначение:** Проверяет наличие свободных мест в группе перед
добавлением нового договора. Если группа заполнена, выбрасывается
исключение.

**Время срабатывания:** BEFORE INSERT
