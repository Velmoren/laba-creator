## 3.1. Анализ и выбор СУБД

Для физической реализации спроектированной базы данных АИС «ЦОД» необходимо выбрать подходящую систему управления базами данных. На рынке представлен широкий спектр реляционных СУБД: Oracle Database, Microsoft SQL Server, MySQL и PostgreSQL.

Коммерческие продукты, такие как Oracle и MS SQL Server, предоставляют мощные инструменты администрирования, встроенные механизмы отказоустойчивости и проприетарные языки (PL/SQL, T-SQL). Однако их лицензирование сопряжено с колоссальными финансовыми затратами, которые зачастую не оправданы для инфраструктуры дата-центров среднего масштаба.

Среди открытых (Open Source) СУБД наибольшей популярностью пользуются MySQL и PostgreSQL. MySQL исторически доминирует в сфере веб-приложений (входит в стек LAMP) благодаря высокой скорости чтения. Однако для сложных корпоративных систем с разветвленной бизнес-логикой на уровне БД (обилием хранимых процедур, сложных транзакций и ограничений целостности) абсолютным лидером является **PostgreSQL**.

PostgreSQL — это объектно-реляционная СУБД с открытым исходным кодом, обладающая рядом критических преимуществ:
1. Строгое соответствие стандартам ANSI SQL.
2. Продвинутый планировщик и оптимизатор запросов, способный эффективно обрабатывать сложные `JOIN`.
3. Поддержка механизма MVCC (Multiversion Concurrency Control), который гарантирует высокую производительность в условиях параллельной работы множества пользователей (чтение не блокирует запись, а запись не блокирует чтение).
4. Мощный язык PL/pgSQL для написания сложной бизнес-логики в триггерах и хранимых процедурах.

На основании вышеизложенного, физическое проектирование и реализация БД АИС «ЦОД» будет производиться в среде **PostgreSQL**. В качестве инструмента администрирования используется графический клиент pgAdmin 4.

## 3.2. Физическое проектирование базы данных (DDL)

В соответствии с даталогической моделью, разработанной во второй главе, был сгенерирован набор скриптов DDL (Data Definition Language) для физического создания объектов базы данных в PostgreSQL. 

Для первичных ключей применен тип `SERIAL` (автоинкрементный integer), который обеспечивает уникальную и последовательную идентификацию каждой записи. Для обеспечения ссылочной целостности заданы внешние ключи (`FOREIGN KEY`) с правилами обновления и удаления (каскадное или ограничивающее).

Ниже приведен фрагмент DDL-скрипта создания ключевых таблиц:

```sql
CREATE TABLE TARIFF (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL UNIQUE,
    power_limit_watts INTEGER NOT NULL,
    bandwidth_mbps INTEGER NOT NULL,
    monthly_cost DECIMAL(10, 2) NOT NULL
);

CREATE TABLE CLIENT (
    id SERIAL PRIMARY KEY,
    company_name VARCHAR(150) NOT NULL,
    inn VARCHAR(12) UNIQUE NOT NULL,
    contact_phone VARCHAR(20),
    email VARCHAR(100)
);

CREATE TABLE CONTRACT (
    id SERIAL PRIMARY KEY,
    client_id INTEGER REFERENCES CLIENT(id) ON DELETE RESTRICT,
    tariff_id INTEGER REFERENCES TARIFF(id) ON DELETE RESTRICT,
    start_date DATE NOT NULL,
    end_date DATE
);

CREATE TABLE DATA_HALL (
    id SERIAL PRIMARY KEY,
    hall_number VARCHAR(10) UNIQUE NOT NULL,
    cooling_system VARCHAR(100),
    max_power_kw INTEGER NOT NULL
);

CREATE TABLE RACK (
    id SERIAL PRIMARY KEY,
    hall_id INTEGER REFERENCES DATA_HALL(id) ON DELETE RESTRICT,
    rack_label VARCHAR(50) UNIQUE NOT NULL,
    capacity_u INTEGER NOT NULL CHECK (capacity_u > 0),
    allocated_power_watts INTEGER NOT NULL
);
```

Для таблиц заданы ограничения целостности: атрибут `NOT NULL` запрещает ввод пустых значений, атрибут `UNIQUE` гарантирует отсутствие дубликатов (например, ИНН клиента или номер стойки должны быть уникальны в пределах всей БД), а `CHECK` накладывает доменные ограничения (вместимость стойки не может быть отрицательной). ## 3.3. Разработка представлений (Views)

Для обеспечения удобства конечных пользователей и сокрытия сложности структуры таблиц (множественных объединений `JOIN`) были разработаны представления. Использование представлений (Views) также повышает уровень безопасности: пользователю можно дать доступ к представлению, запретив прямой доступ к исходным таблицам.

**Представление 1: "Активное_оборудование_клиентов"**
Это представление агрегирует данные об оборудовании, связывая их с контрактами и клиентами. Оно необходимо отделу продаж для генерации счетов. Представление включает использование кириллических псевдонимов столбцов для удобного отображения в интерфейсе.

```sql
CREATE VIEW Активное_оборудование_клиентов AS
SELECT 
    c.company_name AS "Название_компании",
    c.inn AS "ИНН",
    con.id AS "Номер_договора",
    e.eq_type AS "Тип_оборудования",
    e.manufacturer AS "Производитель",
    e.serial_number AS "Серийный_номер",
    e.status AS "Статус"
FROM EQUIPMENT e
JOIN CONTRACT con ON e.contract_id = con.id
JOIN CLIENT c ON con.client_id = c.id
WHERE e.status = 'Активно';
```

**Представление 2: "Статистика_по_стойкам"**
Данное представление используется инженерной службой. Оно группирует данные об установленном оборудовании (через таблицу PLACEMENT) и вычисляет, сколько юнитов в стойке уже занято, а сколько свободно.

```sql
CREATE VIEW Статистика_по_стойкам AS
SELECT 
    r.rack_label AS "Номер_стойки",
    dh.hall_number AS "Машинный_зал",
    r.capacity_u AS "Всего_юнитов",
    COALESCE(SUM(e.size_u), 0) AS "Занято_юнитов",
    r.capacity_u - COALESCE(SUM(e.size_u), 0) AS "Свободно_юнитов"
FROM RACK r
JOIN DATA_HALL dh ON r.hall_id = dh.id
LEFT JOIN PLACEMENT p ON r.id = p.rack_id
LEFT JOIN EQUIPMENT e ON p.equipment_id = e.id
GROUP BY r.id, r.rack_label, dh.hall_number, r.capacity_u;
```

## 3.4. Разработка экранных форм и отчетов

Хотя реализация полноценного клиентского интерфейса (frontend) выходит за рамки проектирования самой БД, в рамках СУБД была спроектирована структура для обеспечения форм и отчетов.

Экранные формы ввода (например, форма регистрации нового клиента или форма открытия инцидента) напрямую маппируются на таблицы `CLIENT` и `INCIDENT`. 

Для генерации документов (отчетов), сопровождающих бизнес-процессы компании ООО «ДатаСфера», используются сложные SQL-запросы с агрегатными функциями. 
В частности, отчет **"Акт о наличии инцидентов за период"**, формируемый для корпоративных клиентов, использует группировку данных по контрактам и фильтрацию по уровню критичности инцидента (severity). Данный отчет эмулирует выгрузку внутреннего документа технической дирекции.

## 3.5. Реализация ограничений и автоматизация бизнес-логики (Триггеры)

Ограничения целостности на уровне первичных и внешних ключей решают не все проблемы бизнес-логики. Существуют правила предметной области, которые требуют сложной алгоритмической проверки перед изменением данных. В PostgreSQL эта задача решается триггерами (Triggers) и триггерными функциями, написанными на языке PL/pgSQL.

В рамках проекта было реализовано три обязательных триггера.

### Триггер 1: Запрет размещения оборудования в занятые юниты (Пространственная коллизия)
Это самый критически важный триггер для дата-центра. При добавлении новой записи в таблицу `PLACEMENT` триггер проверяет, не пересекается ли заданный диапазон юнитов (от `u_start` до `u_end`) с уже установленным оборудованием в данной стойке.

```sql
CREATE OR REPLACE FUNCTION check_rack_collision() RETURNS TRIGGER AS $$
DECLARE
    collision_count INTEGER;
BEGIN
    SELECT COUNT(*) INTO collision_count
    FROM PLACEMENT
    WHERE rack_id = NEW.rack_id
      AND (NEW.u_start <= u_end AND NEW.u_end >= u_start)
      AND id != NEW.id; -- Исключаем самопроверку при UPDATE

    IF collision_count > 0 THEN
        RAISE EXCEPTION 'Конфликт размещения: Юниты с % по % в стойке % уже заняты!', NEW.u_start, NEW.u_end, NEW.rack_id;
    END IF;

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_check_rack_collision
BEFORE INSERT OR UPDATE ON PLACEMENT
FOR EACH ROW EXECUTE FUNCTION check_rack_collision();
```

### Триггер 2: Проверка вместимости стойки
При добавлении оборудования (даже если юниты формально не пересекаются в БД), система должна убедиться, что оборудование физически влезает в стойку (например, `u_end` не может быть 45 в 42-юнитовой стойке).

```sql
CREATE OR REPLACE FUNCTION check_rack_capacity() RETURNS TRIGGER AS $$
DECLARE
    max_u INTEGER;
BEGIN
    SELECT capacity_u INTO max_u FROM RACK WHERE id = NEW.rack_id;
    
    IF NEW.u_end > max_u OR NEW.u_start < 1 THEN
        RAISE EXCEPTION 'Недопустимые юниты: Стойка имеет всего % U', max_u;
    END IF;
    
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_check_rack_capacity
BEFORE INSERT OR UPDATE ON PLACEMENT
FOR EACH ROW EXECUTE FUNCTION check_rack_capacity();
```

### Триггер 3: Защита старых контрактов
Бизнес-правило гласит, что дата окончания контракта не может быть раньше даты его начала. Кроме того, контракты, которые уже завершены, не могут быть удалены из базы (только архивированы), чтобы не нарушить финансовую отчетность.

```sql
CREATE OR REPLACE FUNCTION check_contract_dates() RETURNS TRIGGER AS $$
BEGIN
    IF NEW.end_date IS NOT NULL AND NEW.end_date < NEW.start_date THEN
        RAISE EXCEPTION 'Дата окончания контракта не может быть раньше даты начала';
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_check_contract_dates
BEFORE INSERT OR UPDATE ON CONTRACT
FOR EACH ROW EXECUTE FUNCTION check_contract_dates();
```

## 3.6. Безопасность и контроль доступа

Информационная безопасность — краеугольный камень управления базами данных. Построение системы безопасности в спроектированной БД АИС «ЦОД» реализовано с использованием механизма ролей (Role-Based Access Control, RBAC) СУБД PostgreSQL.

В базе данных созданы три логические роли:
1. `dba_role` — администратор баз данных (Database Administrator). Имеет полные права (`ALL PRIVILEGES`) на модификацию схемы БД (DDL-операции), а также на управление пользователями.
2. `noc_role` — инженер дежурной смены. Имеет право на чтение (`SELECT`), вставку (`INSERT`) и обновление (`UPDATE`) таблиц `INCIDENT`, `EQUIPMENT` и `PLACEMENT`. Право удаления (`DELETE`) для данной роли заблокировано, чтобы инженеры не могли уничтожить историю обслуживания.
3. `sales_role` — менеджер отдела продаж. Имеет право на чтение всех таблиц для формирования отчетов, а также право на редактирование данных в таблицах `CLIENT` и `CONTRACT`.

```sql
CREATE ROLE noc_role;
GRANT SELECT, INSERT, UPDATE ON INCIDENT, EQUIPMENT, PLACEMENT TO noc_role;

CREATE ROLE sales_role;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO sales_role;
GRANT INSERT, UPDATE ON CLIENT, CONTRACT TO sales_role;
```

Кроме того, для критически важных таблиц (таких как `CONTRACT`) рекомендуется в будущем внедрить политику безопасности на уровне строк (Row-Level Security, RLS), чтобы менеджер мог видеть только те контракты, которые заключены с его персональными клиентами.
