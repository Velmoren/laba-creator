## 1. ВВЕДЕНИЕ И ПОСТАНОВКА ЗАДАЧИ
Сегодня я начинаю проект по созданию базы данных для арт-галереи. Задача не просто 'создать таблицы', а заложить фундамент системы, которая выдержит тысячи запросов. Основной упор — на DDL (Data Definition Language) и управление первичными ключами. 

## 2. РЕАЛИЗАЦИЯ СУРРОГАТНЫХ КЛЮЧЕЙ
В Postgres есть тип SERIAL, но в промышленной разработке чаще используют SEQUENCE. Это дает полный контроль над инкрементом. Я создаю три последовательности для каждой сущности. 
Для покупателей (CUSTOMER) я устанавливаю начало с 5000, чтобы не путать их с ID художников.

```sql
CREATE SEQUENCE ArtistID_Seq START WITH 1;
CREATE SEQUENCE WorkID_Seq START WITH 1;
CREATE SEQUENCE CustID_Seq START WITH 5000;
```

## 3. СОЗДАНИЕ ТАБЛИЦ И ПЕРВИЧНЫХ КЛЮЧЕЙ
Согласно методичке, я должен показать два способа задания Primary Key. 
Для таблицы ARTIST я задал ключ прямо в определении столбца (inline). Для таблицы CUSTOMER я сначала создал структуру, а затем навесил ключ через ALTER TABLE. Это позволяет именовать ограничение (pk_customer_id), что сильно упрощает жизнь при анализе ошибок.

```sql
CREATE TABLE ARTIST (
    ArtistID int PRIMARY KEY DEFAULT nextval('ArtistID_Seq'),
    Name varchar(255) NOT NULL,
    Nationality varchar(100)
);

CREATE TABLE CUSTOMER (
    CustomerID int DEFAULT nextval('CustID_Seq'),
    Name varchar(255) NOT NULL,
    AreaCode varchar(10),
    PhoneNumber varchar(50)
);
ALTER TABLE CUSTOMER ADD CONSTRAINT pk_customer_id PRIMARY KEY (CustomerID);
```

## 4. ВЫЧИСЛЕНИЯ И ВЫВОДЫ
Я проверил работу: выполнил INSERT в CUSTOMER. Запись получила ID 5000. Вторая — 5001. Это подтверждает, что последовательность CustID_Seq работает корректно. Использование именованных ключей через ALTER TABLE — мой выбор на будущее.
