# Глава 1. Системный анализ предметной области

## 1.1. Анализ объекта автоматизации

### 1.1.1. Общие сведения об объекте автоматизации

Объектом автоматизации является образовательный центр «Компьютерные курсы», предоставляющий услуги дополнительного профессионального образования в сфере информационных технологий. Центр работает на рынке образовательных услуг более 5 лет и специализируется на проведении курсов для взрослых.

**Основные направления деятельности центра:**
- Курсы программирования (Python, Java, JavaScript, C++)
- Курсы администрирования (Linux, Windows Server, сетевые технологии)
- Курсы веб-разработки (HTML/CSS, React, Angular)
- Базы данных и SQL
- Курсы для начинающих (основы ПК, офисные приложения)

**Организационно-правовая форма:** Общество с ограниченной ответственностью (ООО «Компьютерные курсы»)

**Штатная структура:**
- Администрация (директор, заместитель директора, бухгалтер)
- Преподавательский состав (10-15 преподавателей)
- Менеджеры по работе с клиентами (3-4 человека)
- Технический персонал (2 человека)

### 1.1.2. Организационно-функциональная структура

Для эффективного функционирования образовательного центра необходимо определить структуру подразделений и их взаимосвязи. Каждое подразделение выполняет определённые функции, которые обеспечивают бесперебойную работу всего центра. Организационная структура определяет распределение responsibilities между сотрудниками и порядок взаимодействия подразделений. Рассмотрим основные подразделения образовательного центра и их функции в таблице ниже.

**Подразделения предприятия и их функции:**

| Подразделение | Основные функции |
|---------------|------------------|
| Администрация | Общее руководство, финансовый учёт, стратегическое планирование |
| Учебный отдел | Формирование расписания, управление курсами, контроль качества обучения |
| Отдел продаж | Приём заявок, консультирование клиентов, заключение договоров |
| Преподаватели | Проведение занятий, проверка домашних заданий, ведение журнала успеваемости |
| Бухгалтерия | Расчёт зарплаты, учёт оплат, формирование отчётности |

**Взаимосвязи между подразделениями:**
- Отдел продаж передаёт заявки в учебный отдел для формирования групп
- Учебный отдел направляет преподавателям расписание занятий
- Бухгалтерия получает данные об оплатах от отдела продаж
- Администрация контролирует работу всех подразделений

### 1.1.3. Основные бизнес-процессы

**Бизнес-процессы образовательного центра:**

1. **Приём и обработка заявок**
   - Клиент оставляет заявку на курс
   - Менеджер обрабатывает заявку
   - Клиент заключает договор

2. **Формирование учебных групп**
   - Набор группы (минимум 5, максимум 15 человек)
   - Определение расписания
   - Назначение преподавателя

3. **Учебный процесс**
   - Проведение занятий согласно расписанию
   - Ведение журнала посещаемости и успеваемости
   - Выдача домашних заданий и их проверка

4. **Завершение обучения**
   - Итоговое тестирование
   - Выдача сертификатов
   - Сбор отзывов

5. **Финансовый учёт**
   - Регистрация платежей
   - Расчёт стоимости обучения
   - Формирование финансовых отчётов

### 1.1.4. Информационные объекты

**Данные, обрабатываемые в системе:**

- Информация о клиентах (слушателях)
- Данные о курсах и программах обучения
- Сведения о преподавателях
- Расписание занятий
- Успеваемость и посещаемость
- Финансовые данные (оплаты, задолженности)
- Документы (договоры, сертификаты)

**Источники информации:**
- Заявки клиентов
- Договоры
- Результаты тестирования
- Платёжные документы

**Потребители информации:**
- Администрация (отчёты, статистика)
- Преподаватели (списки групп, расписание)
- Клиенты (информация о курсах, успеваемости)

### 1.1.5. Группы пользователей БД

Для обеспечения безопасности и разграничения доступа к данным необходимо определить группы пользователей системы и их права. Каждая группа характеризуется определённым набором операций, которые доступны пользователям данной роли. Это позволяет контролировать доступ к конфиденциальной информации и обеспечить целостность данных. Разграничение прав доступа является важным аспектом проектирования любой информационной системы, особенно в образовательной сфере, где хранятся персональные данные клиентов. В таблице представлены основные группы пользователей и их полномочия.

| Группа пользователей | Описание | Основные операции |
|----------------------|----------|-------------------|
| Администратор | Полный доступ к системе | Все операции, управление пользователями |
| Менеджер | Работа с клиентами и заявками | Добавление/редактирование клиентов, заявок, договоров |
| Преподаватель | Ведение учебного процесса | Просмотр расписания, ввод оценок, ведение журнала |
| Бухгалтер | Финансовый учёт | Добавление платежей, формирование отчётов |
| Клиент | Ограниченный просмотр | Просмотр своих курсов, успеваемости |

### 1.1.6. Входные и выходные документы

**Входные документы:**
- Заявка на обучение
- Договор на оказание образовательных услуг
- Платёжный документ (квитанция, чек)
- Результаты входного тестирования

**Выходные документы:**
- Расписание занятий
- Список группы
- Журнал посещаемости
- Ведомость успеваемости
- Сертификат о прохождении курса
- Финансовый отчёт

## 1.2. Обзор информационных технологий

### 1.2.1. CASE-средства для проектирования БД

Для проектирования базы данных могут использоваться специализированные CASE-средства (Computer-Aided Software Engineering), которые позволяют автоматизировать процесс разработки моделей данных [2, 10]. Рассмотрим наиболее популярные инструменты:

**ERwin Data Modeler** — один из наиболее популярных инструментов для проектирования баз данных. Позволяет создавать инфологические и даталогические модели, генерировать SQL-скрипты для различных СУБД, проводить прямой и обратный инжиниринг.

**PowerDesigner** — мощное средство моделирования данных от SAP, поддерживающее различные нотации и позволяющее работать с большими проектами.

**DBDesigner** — бесплатное средство для визуального проектирования баз данных.

**pgModeler** — специализированное средство для проектирования баз данных PostgreSQL с открытым исходным кодом.

Для данного проекта рекомендуется использовать **ERwin Data Modeler** благодаря широким возможностям и поддержке нотации Crow's Foot, которая наиболее наглядно отражает связи между сущностями.

### 1.2.2. Классы СУБД

Рассмотрим основные классы систем управления базами данных, подробно описанные в литературе [3, 9]:

**Иерархические СУБД** — организуют данные в виде дерева с отношениями «родитель-потомок». Применяются в специализированных системах (файловые системы, XML-базы). Примеры: IBM IMS, Windows Registry.

**Сетевые СУБД** — развитие иерархической модели, позволяют связи «многие ко многим» через сетевые структуры. Используются в системах с сложными взаимосвязями. Примеры: ADABAS, Cache.

**Реляционные СУБД** — наиболее распространённый тип, организует данные в таблицах со связями через общие ключи. Преимущества: теоретическая база (реляционная алгебра), гибкость запросов, стандартный язык SQL. Примеры: PostgreSQL, MySQL, Oracle, MS SQL Server.

**Объектно-ориентированные СУБД** — хранят данные в виде объектов, как в ООП. Применяются для сложных структур данных. Примеры: ObjectDB, db4o.

Для проектирования базы данных образовательного центра наиболее подходит **реляционная модель данных**, поскольку она:
- Обладает развитым математическим аппаратом
- Поддерживается всеми современными СУБД
- Позволяет эффективно обрабатывать структурированные данные
- Имеет стандартизированный язык запросов SQL

### 1.2.3. Выбор СУБД

Для реализации базы данных необходимо выбрать подходящую систему управления базами данных. Рассмотрим наиболее популярные СУБД, используемые для разработки информационных систем, и проведём их сравнительный анализ по основным критериям: стоимость, функциональность, масштабируемость и область применения. Выбор СУБД является важным этапом проектирования, поскольку от него зависит производительность, надёжность и безопасность системы. Каждая СУБД имеет свои преимущества и недостатки, которые необходимо учитывать при принятии решения. В таблице представлено сравнение наиболее распространённых СУБД.

| СУБД | Преимущества | Недостатки | Применение |
|------|--------------|------------|------------|
| PostgreSQL | Открытый код, мощные функции, расширяемость | Сложнее в настройке | Средние и крупные проекты |
| MySQL | Простота, скорость, бесплатность | Ограниченная функциональность | Веб-приложения |
| Oracle | Масштабируемость, надёжность | Высокая стоимость | Корпоративные системы |
| MS SQL Server | Интеграция с Windows, инструменты | Только Windows | Корпоративные решения |

Для данного проекта выбрана **СУБД PostgreSQL** [7, 16] по следующим причинам:
1. Открытая лицензия (бесплатное использование)
2. Полное соответствие требованиям задания (не Access)
3. Поддержка сложных запросов, триггеров, представлений
4. Широкие возможности обеспечения безопасности
5. Активное сообщество и документация на русском языке

## 1.3. Обзор продуктов-аналогов

### 1.3.1. Обзор существующих систем

На рынке представлено несколько систем управления образовательным процессом:

**1С:Образование** — комплексное решение для автоматизации образовательных учреждений. Включает модули для учёта учащихся, формирования расписания, ведения электронного журнала.

*Преимущества:* Интеграция с другими продуктами 1С, широкий функционал
*Недостатки:* Высокая стоимость лицензии, сложность настройки

**NetSchool** — система управления образовательным процессом для школ и колледжей.

*Преимущества:* Специализация на школьном образовании, электронный журнал
*Недостатки:* Не адаптирована для коммерческих курсов

**Docebo** — облачная LMS (Learning Management System) для управления обучением.

*Преимущества:* Облачный сервис, современный интерфейс, мобильность
*Недостатки:* Оплата за пользователя, ограниченная локализация

**Google Workspace for Education** — набор инструментов для образовательных учреждений.

*Преимущества:* Бесплатность для образования, интеграция с Google
*Недостатки:* Не полноценная система управления курсами

На основании проведённого анализа существующих систем можно сделать вывод, что большинство решений либо слишком дороги для небольших образовательных центров, либо не соответствуют специфике краткосрочных компьютерных курсов.

Разрабатываемая система «Компьютерные курсы» будет отличаться от аналогов:
1. **Оптимизация под специфику курсов** — учёт продолжительности курсов, интенсивности, периодичности занятий
2. **Доступная стоимость** — бесплатная СУБД PostgreSQL
3. **Гибкая настройка** — возможность адаптации под конкретные нужды
4. **Простота использования** — интуитивный интерфейс для менеджеров

## 1.4. Требования к разрабатываемой базе данных

### 1.4.1. Функциональные требования

Согласно ГОСТ 34.601-90 [1], разрабатываемая система должна обеспечивать:

**Функции учёта:**
- Регистрация и хранение информации о клиентах
- Учёт курсов и программ обучения
- Ведение данных о преподавателях
- Регистрация учебных групп
- Учёт посещаемости и успеваемости

**Функции управления:**
- Формирование расписания занятий
- Автоматический расчёт стоимости обучения
- Контроль оплат и задолженностей

**Функции отчётности:**
- Формирование списков групп
- Создание ведомостей успеваемости
- Генерация финансовых отчётов
- Создание отчётов о посещаемости

### 1.4.2. Требования к данным

**Сущности предметной области:**

1. Клиенты (слушатели) — персональные данные, контакты
2. Курсы — название, описание, продолжительность, стоимость
3. Группы — наименование, курс, преподаватель, расписание
4. Преподаватели — персональные данные, квалификация, контакты
5. Занятия — дата, время, аудитория, тема занятия
6. Посещаемость — статус присутствия, причина отсутствия
7. Успеваемость — оценки, домашние задания
8. Договоры — номер, дата, условия
9. Оплаты — дата, сумма, статус
10. Сертификаты — номер, дата выдачи, оценка

### 1.4.3. Выходные документы

Разрабатываемая система должна формировать различные отчёты и документы, необходимые для управления образовательным центром. Выходные документы обеспечивают информационную поддержку принятия управленческих решений и служат для документирования результатов учебного процесса. Отчёты позволяют анализировать эффективность работы центра, отслеживать финансовые показатели и контролировать качество обучения. Формирование документов должно быть автоматизировано для снижения ручного труда сотрудников. В таблице представлен перечень основных выходных документов.

| Отчёт | Назначение | Периодичность |
|-------|------------|---------------|
| Список группы | Перечень слушателей группы | По запросу |
| Расписание занятий | План занятий на период | Еженедельно |
| Ведомость успеваемости | Оценки и посещаемость | По окончании курса |
| Финансовый отчёт | Приход/расход | Ежемесячно |
| Отчёт о задолженностях | Просроченные платежи | Еженедельно |

### 1.4.4. Требования к безопасности

- Разграничение доступа по ролям пользователей
- Парольная защита учётных записей
- Журналирование действий пользователей
- Резервное копирование данных
