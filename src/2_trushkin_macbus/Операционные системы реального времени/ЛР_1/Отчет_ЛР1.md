## 1. Введение и теоретические сведения

Операционная система FreeBSD базируется на принципах UNIX, где «всё есть файл». Одной из ключевых особенностей FreeBSD является строгое следование иерархии файловой системы, описанной в `man hier`. В отличие от Linux, во FreeBSD системные файлы и пользовательские приложения четко разделены: базовая система находится в `/bin`, `/sbin`, `/usr/bin`, а стороннее ПО из портов или пакетов устанавливается исключительно в `/usr/local/bin`, `/usr/local/etc` и т.д.

### 1.1. Иерархия каталогов
- `/` — корневой каталог.
- `/bin` — основные утилиты командной оболочки.
- `/boot` — ядро (kernel) и загрузчик.
- `/etc` — конфигурационные файлы системы.
- `/home` — домашние каталоги пользователей (в FreeBSD часто является ссылкой на `/usr/home`).
- `/root` — домашний каталог суперпользователя.

### 1.2. Редакторы текста: vim и nano
Для настройки системы крайне важно владеть консольными редакторами. `vi` (или его улучшенная версия `vim`) является стандартом де-факто в BSD-системах. Он имеет модальную архитектуру: режим команд, режим вставки и режим последней строки. Это позволяет эффективно манипулировать текстом без использования мыши.

### 1.3. Ссылки: Жесткие (Hard Links) и Символические (Soft/Symbolic Links)
- **Жесткая ссылка (`ln`):** Создает дополнительное имя для того же inode (индекса файла на диске). Удаление оригинального имени не приводит к потере данных, пока существует хотя бы одна жесткая ссылка. Ограничение: нельзя создавать между разными файловыми системами и для каталогов.
- **Символическая ссылка (`ln -s`):** Специальный файл, содержащий путь к другому файлу. Если оригинальный файл удалить, ссылка станет «битой» (dangling link).

## 2. Ход работы

### 2.1. Подготовка структуры каталогов
Для начала я создал дерево каталогов, имитирующее структуру проекта в моей домашней директории. Я использовал команду `mkdir` с ключом `-p` для рекурсивного создания путей.

```bash
trushkin@macbus $ mkdir -p ~/lab1/project/src ~/lab1/project/bin ~/lab1/project/docs
trushkin@macbus $ cd ~/lab1/project
```

### 2.2. Работа с текстовыми редакторами
Я создал файл `README.md` в каталоге `docs`, используя редактор `vim`. В нем я описал назначение структуры.

```bash
trushkin@macbus $ vim docs/README.md
```
*Внутри vim я нажал 'i' для перехода в режим вставки, ввел текст, нажал 'Esc', затем ':wq' для сохранения и выхода.*

Затем я создал файл `notes.txt` с помощью `nano`, чтобы сравнить удобство работы.

```bash
trushkin@macbus $ nano docs/notes.txt
```

### 2.3. Создание и проверка ссылок
Создадим файл `source.txt` и сделаем на него ссылки двух типов.

```bash
trushkin@macbus $ echo "Original data" > docs/source.txt
trushkin@macbus $ ln docs/source.txt docs/hardlink_to_source
trushkin@macbus $ ln -s docs/source.txt docs/softlink_to_source
```

Проверим структуру и inode файлов:
```bash
trushkin@macbus $ ls -li docs/
```
Я заметил, что `source.txt` и `hardlink_to_source` имеют одинаковый номер inode, а `softlink_to_source` — другой, и его размер соответствует длине пути к оригиналу.

### 2.4. Тестирование целостности ссылок
Удалим оригинальный файл и посмотрим, что произойдет.

```bash
trushkin@macbus $ rm docs/source.txt
trushkin@macbus $ cat docs/hardlink_to_source
# Вывод: Original data (данные доступны)
trushkin@macbus $ cat docs/softlink_to_source
# Вывод: cat: docs/softlink_to_source: No such file or directory (ссылка сломана)
```

В ходе выполнения лабораторной работы №1 я закрепил навыки работы в терминале FreeBSD. Я изучил основные команды навигации и манипуляции файлами (`mkdir`, `cd`, `ls`, `rm`). Практический опыт показал, что жесткие ссылки более надежны для сохранения доступа к данным при переименовании или удалении оригинальных имен, однако символические ссылки обладают большей гибкостью, позволяя ссылаться на каталоги и объекты на других разделах диска. Также я освоил базовые приемы работы в редакторе `vim`, что является необходимым навыком для любого системного администратора FreeBSD.
