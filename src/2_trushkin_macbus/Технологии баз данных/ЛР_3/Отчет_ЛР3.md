# ОТЧЕТ ПО ЛАБОРАТОРНОЙ РАБОТЕ №3: КОМПЛЕКСНОЕ ПРОЕКТИРОВАНИЕ И ФИЗИЧЕСКАЯ РЕАЛИЗАЦИЯ РЕЛЯЦИОННОЙ МОДЕЛИ ДАННЫХ

## ВВЕДЕНИЕ
В современной разработке информационных систем этап проектирования базы данных является критическим. Качество спроектированной схемы напрямую влияет на производительность, масштабируемость и целостность данных в будущем. В рамках данной лабораторной работы исследуются методы реализации физической схемы данных для художественной галереи «View Ridge Gallery» средствами объектно-реляционной СУБД PostgreSQL. Основной акцент сделан на использовании языка описания данных (DDL), механизмах управления первичными ключами и автоматизации генерации суррогатных идентификаторов.

## 1. ПОСТАНОВКА ЗАДАЧИ
Требуется разработать базу данных, способную эффективно хранить информацию о деятельности галереи. Система должна поддерживать учет художников, их произведений, данных о покупателях, а также детальную историю транзакций. Ключевым требованием является демонстрация различных подходов к обеспечению уникальности записей, включая использование встроенных последовательностей PostgreSQL для формирования суррогатных ключей.

## 2. ТЕОРЕТИЧЕСКИЕ ОСНОВЫ DDL
Язык DDL (Data Definition Language) включает в себя операторы для создания (CREATE), модификации (ALTER) и удаления (DROP) структур данных. В PostgreSQL первичный ключ (PRIMARY KEY) обеспечивает уникальность строк и автоматически создает индекс. Мы рассмотрим два метода декларации ключей:
1. **Инлайновый метод:** PK определяется непосредственно в инструкции CREATE TABLE. Это удобно для простых, компактных схем.
2. **Метод модификации:** PK добавляется через инструкцию ALTER TABLE. Этот подход является более гибким, так как позволяет именовать ограничения (Constraints) согласно корпоративным стандартам, что облегчает последующее администрирование и отладку.

## 3. ХОД РАБОТЫ
### 3.1. Создание последовательностей
Для реализации независимых суррогатных ключей, которые не привязаны жестко к одной таблице, мы создаем отдельные объекты SEQUENCE. Это позволяет, например, использовать одну последовательность для группы связанных сущностей или задавать специфические правила инкремента.

```sql
-- Последовательность для идентификаторов художников
CREATE SEQUENCE ArtistID_Seq START WITH 1 INCREMENT BY 1;

-- Последовательность для учета произведений
CREATE SEQUENCE WorkID_Seq START WITH 1 INCREMENT BY 1;

-- Последовательность для клиентов с начальным значением 1000 для отличия от внутренних ID
CREATE SEQUENCE CustID_Seq START WITH 1000 INCREMENT BY 1;
```

### 3.2. Реализация таблиц и управление ключами
Мы приступаем к созданию таблиц, используя комбинированный подход к заданию ограничений.

#### Таблица ARTIST (Художники)
Здесь мы используем первый способ задания PK — непосредственно в столбце. Поле `BirthDate` выбрано как целочисленное для хранения года рождения, что упрощает математические расчеты возраста без лишних накладных расходов типа `date`.

```sql
CREATE TABLE ARTIST (
    ArtistID int PRIMARY KEY DEFAULT nextval('ArtistID_Seq'),
    Name varchar(255) NOT NULL,
    Nationality varchar(100) DEFAULT 'Unknown',
    BirthDate int,
    DeceasedDate int
);
```

#### Таблица CUSTOMER (Покупатели)
Для этой таблицы мы применим второй способ — через модификацию. Это позволит нам явно задать имя ограничения `PK_CUSTOMER`, что соответствует академическим стандартам оформления БД.

```sql
CREATE TABLE CUSTOMER (
    CustomerID int DEFAULT nextval('CustID_Seq'),
    Name varchar(255) NOT NULL,
    AreaCode varchar(10),
    PhoneNumber varchar(50)
);

-- Явное добавление первичного ключа
ALTER TABLE CUSTOMER ADD CONSTRAINT PK_CUSTOMER PRIMARY KEY (CustomerID);
```

### 3.3. Таблица связи CUSTOMER_ARTIST_INT
Эта таблица реализует связь "многие-ко-многим", позволяя фиксировать интерес нескольких клиентов к нескольким художникам. Здесь мы используем композитный первичный ключ.

```sql
CREATE TABLE CUSTOMER_ARTIST_INT (
    ArtistID int NOT NULL,
    CustomerID int NOT NULL,
    CONSTRAINT PK_CUST_ART PRIMARY KEY (ArtistID, CustomerID)
);
```
