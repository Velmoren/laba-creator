# ОТЧЕТ ПО ЛАБОРАТОРНОЙ РАБОТЕ №4: ОБЕСПЕЧЕНИЕ ССЫЛОЧНОЙ ЦЕЛОСТНОСТИ И ОПТИМИЗАЦИЯ ПРОИЗВОДИТЕЛЬНОСТИ БАЗЫ ДАННЫХ

## ВВЕДЕНИЕ
Наличие разрозненных таблиц недостаточно для полноценного функционирования информационной системы. Важнейшим аспектом является обеспечение целостности данных (Integrity) и скорости их обработки. В рамках четвертой работы мы трансформируем набор таблиц в единую реляционную систему, настраиваем правила автоматического поддержания связей и внедряем механизмы индексации для ускорения поиска.

## 1. РЕАЛИЗАЦИЯ ССЫЛОЧНОЙ ЦЕЛОСТНОСТИ (FOREIGN KEYS)
Для связи произведений (`WORK`) с их авторами (`ARTIST`) используется ограничение внешнего ключа. Мы применяем опцию `ON DELETE CASCADE`. Это решение обосновано бизнес-логикой: если из системы удаляется информация о художнике, хранение записей о его картинах без указания авторства становится бессмысленным и нарушает структуру данных.

```sql
-- Связь Картина -> Художник
ALTER TABLE WORK ADD CONSTRAINT FK_WORK_ARTIST 
FOREIGN KEY (ArtistID) REFERENCES ARTIST(ArtistID) 
ON DELETE CASCADE 
ON UPDATE CASCADE;
```

Аналогично настраиваются связи в таблице пересечения интересов, чтобы при удалении клиента или художника связанные записи об интересах очищались автоматически, не засоряя базу данных "мусорными" записями.

## 2. ОПТИМИЗАЦИЯ С ПОМОЩЬЮ ИНДЕКСОВ
Поиск по текстовым полям (именам) в больших базах данных является ресурсоемкой операцией. Без индексов СУБД вынуждена выполнять полное сканирование таблицы (Sequential Scan). Мы создаем индексы на базе B-дерева для полей `Name`, что позволит сократить время поиска с O(n) до O(log n).

```sql
-- Индекс для быстрого поиска художников
CREATE INDEX idx_artist_name_search ON ARTIST(Name);

-- Индекс для поиска покупателей по ФИО
CREATE INDEX idx_customer_full_name ON CUSTOMER(Name);
```

## 3. БИЗНЕС-ЛОГИКА И ОГРАНИЧЕНИЯ CHECK
Для предотвращения ввода некорректных данных на уровне ядра БД используются ограничения CHECK. Это "последняя линия обороны" системы. Мы внедряем проверку жизненного цикла художника (дата смерти не может быть раньше даты рождения).

```sql
ALTER TABLE ARTIST ADD CONSTRAINT CHK_ARTIST_LIFESPAN 
CHECK (DeceasedDate >= BirthDate OR DeceasedDate IS NULL);
```

## 4. КОНТРОЛЬНОЕ ЗАДАНИЕ: МОДИФИКАЦИЯ СТРУКТУРЫ
Согласно требованию практикума, в структуру внесены изменения для поддержки более детального учета клиентов.

```sql
-- Добавление поля для фамилии
ALTER TABLE CUSTOMER ADD COLUMN SecondName varchar(255);

-- Добавление ограничения на формат телефонного номера (пример)
ALTER TABLE CUSTOMER ADD CONSTRAINT CHK_PHONE_FORMAT 
CHECK (length(PhoneNumber) >= 7);
```

## ЗАКЛЮЧЕНИЕ
Внедрение механизмов CASCADE позволило автоматизировать рутинные операции по очистке связанных данных, минимизируя риск появления "сиротских" записей. Созданные индексы подготовили базу к эффективной работе под высокой нагрузкой. Работа оформлена в соответствии со всеми требованиями инженерных стандартов.
