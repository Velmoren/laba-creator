## 2.1. Разработка инфологической модели (ER-модели)

**Цель:** Создать концептуальную модель данных предметной области

### Требования к модели:

- **Не менее 10 сущностей**
- **1-2 связи «многие ко многим»**
- ER-диаграмма в графической нотации (Crow's Foot, Chen и др.)

### Что описать:

1. **Сущности и их атрибуты**
   - Перечислить все сущности
   - Описать атрибуты каждой сущности
   - Указать ключевые атрибуты

2. **Связи между сущностями**
   - Тип связи (1:1, 1:N, M:N)
   - Описать каждую связь
   - Обосновать необходимость связей

3. **ER-диаграмма**
   - Рисунок с изображением модели
   - Использовать стандартную нотацию

### Примеры сущностей:

```
Сущность: Клиент
├── id_клиента (PK) — первичный ключ
├── ФИО
├── Адрес
├── Телефон
├── Email
└── Дата регистрации

Сущность: Заказ
├── id_заказа (PK)
├── Дата заказа
├── Сумма
├── Статус
└── id_клиента (FK)
```

### Нотация Crow's Foot:

```
┌─────────┐          ┌─────────┐
│ Клиент  │───1,N─────│  Заказ  │
├─────────┤          ├─────────┤
│ PK,FK   │          │ PK      │
│ Атрибуты│          │ Атрибуты│
└─────────┘          └─────────┘
```

## 2.2. Обоснование выбора модели данных

**Цель:** Обосновать выбор типа модели данных

### Модели данных:

| Модель | Описание | Когда использовать |
|--------|----------|-------------------|
| Иерархическая | Древовидная структура | Иерархические данные |
| Сетевая | Графовая структура | Сложные взаимосвязи |
| **Реляционная** | Таблицы со связями | Универсальная, **рекомендуется** |
| Объектно-ориентированная | Объекты с методами | Сложные структуры |

### Что написать:

```
Для разрабатываемой базы данных выбрана реляционная модель данных
по следующим причинам:
1. ...
2. ...
3. ...
```

## 2.3. Даталогическое проектирование

**Цель:** Перейти от ER-модели к реляционной схеме

### Правила перехода:

1. **Каждая сущность** → таблица
2. **Атрибуты сущности** → столбцы таблицы
3. **Первичный ключ** → PRIMARY KEY
4. **Связи 1:N** → внешний ключ (FOREIGN KEY) в дочерней таблице
5. **Связи M:N** → промежуточная таблица

### Пример перехода:

```
ER-модель: Клиент 1,N — N,1 Заказ

Реляционная модель:

Таблица Клиент:
- id_клиент (PK)
- ФИО
- Адрес
- Телефон

Таблица Заказ:
- id_заказ (PK)
- Дата
- Сумма
- id_клиент (FK) → ссылка на Клиент.id_клиент
```

### Что описать:

1. **Таблицы с атрибутами**
   - Имена таблиц
   - Столбцы и типы данных
   - Ограничения

2. **Схема базы данных**
   - Рисунок со связями между таблицами
   - Обозначения ключей

3. **SQL-скрипт создания таблиц** (можно в приложении)

## 2.4. Нормализация схемы БД

**Цель:** Привести схему к нормальным формам

### Нормальные формы:

| Форма | Требование |
|-------|------------|
| 1НФ | Атомарные значения, нет повторяющихся групп |
| 2НФ | 1НФ + нет частичных зависимостей от ключа |
| 3НФ | 2НФ + нет транзитивных зависимостей |
| НФБК | 3НФ + нет зависимостей от ключа, кроме полной |
| 4НФ | НФБК + нет многозначных зависимостей |
| 5НФ | 4НФ + нет зависимостей соединения |

### Что описать:

1. **К какой нормальной форме приведена схема**
   - Обычно достаточно 3НФ

2. **Процесс нормализации**
   - Какие аномалии были устранены
   - Как изменялась схема

3. **Примеры:**
```
Исходная таблица (1НФ):
┌─────────┬───────────────┬─────────────┐
│ Заказ   │ Список товаров │ Количество   │
├─────────┼───────────────┼─────────────┤
│ 1       │ Товар1,Товар2 │ 10, 20      │ ← НЕ атомарно!
└─────────┴───────────────┴─────────────┘

После нормализации (2НФ/3НФ):
┌─────────┐
│ Заказ   │
├─────────┤
│ id_заказ│
│ Дата    │
└─────────┘
     │
     │ 1,N
     ▼
┌─────────┬─────────────┐
│ Заказ-Товар           │
├─────────┼─────────────┤
│ id_заказ│ ← FK        │
│ id_товар│ ← FK        │
│ Количество            │
└─────────┴─────────────┘
```
